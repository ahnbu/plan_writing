# Why We Need 'History Keeper'?

> **Subtitle**: 단순 자동화를 넘어선 '사고하는 파트너(Thinking Partner)'를 위한 설계 철학

## 1. 도입 배경

AI 에이전트가 복잡한 전략 수립 업무를 수행할 때, 사용자는 종종 **"블랙박스(Black Box)"** 문제에 직면합니다.
_"결과물이 나왔는데, 왜 A가 아니라 B로 결정된 거지?"_, _"중간에 어떤 근거로 논리가 바뀐 거지?"_
이 과정을 투명하게 드러내고 관리하기 위해 History Keeper가 도입되었습니다.

## 2. 핵심 가치 (Core Value Proposition)

### ① 블랙박스의 해소 (Transparency)

- **Problem**: `ghost_deck_v1`과 `v2`의 차이를 일일이 대조하기 어렵습니다.
- **Solution**: *"시장 성장률 데이터가 5%가 아닌 2%로 확인되어, 낙관적 시나리오를 폐기하고 보수적 시나리오로 전략을 수정했음"*과 같이 **변경의 이유**를 명시적으로 기록합니다.
- **Effect**: 결과물에 대한 **신뢰도(Reliability)**가 상승합니다.

### ② 전략적 복기 (Process Learning)

- **Problem**: 프로젝트가 끝나면 결과물만 남고, 과정에서의 배움은 휘발됩니다.
- **Solution**: 프로젝트 종료 후 **"우리의 초기 가설이 어디서 틀렸는가?"**를 복기할 수 있는 **사고의 궤적(Log)**을 남깁니다.
- **Effect**: 사용자(전략가)의 기획력 향상을 돕는 **메타 인지(Meta-cognition)** 도구가 됩니다.

### ③ 디버깅 및 품질 관리 (Debugging)

- **Problem**: 최종 결과물이 미흡할 때, 원인을 찾기 위해 전체를 다시 실행해야 합니다.
- **Solution**: 어느 단계(예: 리서치 단계)에서 잘못된 데이터가 유입되었는지 기록을 통해 즉시 추적 가능합니다.
- **Effect**: 프롬프트 튜닝 및 프로세스 개선이 정교해집니다.

## 3. Trade-off 분석

| 구분       | History Keeper 없음         | History Keeper 있음                    |
| :--------- | :-------------------------- | :------------------------------------- |
| **속도**   | 빠름 (최적화 중심)          | 다소 느림 (단계별 메타 분석 시간 추가) |
| **결과물** | `Final Report` Only         | `Final Report` + `Workflow History`    |
| **지향점** | 결과 중심 (Result-oriented) | **과정 중심 (Process-oriented)**       |

## 4. 결론

**"단순히 글을 쓰는 기계가 아니라, 함께 고민하는 파트너를 원한다면 History Keeper는 필수적입니다."**
이 기능은 AI의 사고 과정을 인간이 감독하고 승인할 수 있게 만드는 **협업의 연결고리(Interface)**입니다.
